│ Refactor Plan: Decouple Frontend from Backend                                   │
│                                                                                 │
│ Context                                                                         │
│                                                                                 │
│ The app works but has structural issues accumulated over time:                  │
│ - Duplicated processing logic — the same skrutable calls exist 3 times: index() │
│  POST, whole_file(), and /api/* routes                                          │
│ - Session abuse — Flask session stores UI state (action, schemes, checkboxes,   │
│ settings) that should just be sent with each request. Two API endpoints         │
│ (/api/transliterate, /api/split) have real bugs from reading session vars that  │
│ API consumers never set                                                         │
│ - flask.g as glue — g.text_input/g.text_output shuttle data between             │
│ process_form() and the action handlers, but this is just passing function       │
│ arguments with extra steps                                                      │
│ - Full-page form POSTs — every action reloads the entire page; the frontend     │
│ can't be redesigned independently of the Python code                            │
│                                                                                 │
│ The goal: make the frontend call API endpoints via JS (like the OCR page        │
│ already does), so the UI is decoupled from the backend and ready for a future   │
│ redesign.                                                                       │
│                                                                                 │
│ Constraints                                                                     │
│                                                                                 │
│ - Same inputs must produce same outputs — no processing changes                 │
│ - All existing pages/resources remain available                                 │
│ - 128 MB upload support preserved (CustomRequest/MAX_CONTENT_LENGTH stay)       │
│ - External API consumers (plain text .text responses) must not break            │
│ - App stays in a single flask_app.py — no blueprints                            │
│                                                                                 │
│ Decision                                                                        │
│                                                                                 │
│ - API format: Use Accept: application/json header to content-negotiate.         │
│ Frontend sends this header and gets JSON; existing consumers get plain text as  │
│ before.                                                                         │
│                                                                                 │

│ Risk notes                                                                      │
│                                                                                 │
│ - Large text via fetch(): fetch() sends FormData, which hits the same           │
│ CustomRequest/MAX_CONTENT_LENGTH code path as form POSTs. The 128 MB limit is   │
│ enforced at the Werkzeug/Flask level regardless of how the request arrives. No  │
│ change needed.                                                                  │
│ - Backward compat: Accept header negotiation means existing API consumers are   │
│ unaffected. Example URLs redirect. Reset still works.                           │
│ - Incremental: Each phase is independently deployable. The app works after      │
│ every step.                                                                     │
